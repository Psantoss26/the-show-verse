El trabajo ha demostrado que los objetivos planteados son alcanzables con las tecnologías actuales de desarrollo web y sin necesidad de infraestructura de servidor propia. Los resultados obtenidos permiten extraer las siguientes conclusiones.

En primer lugar, la arquitectura Next.js App Router con React Server Components ha demostrado ser particularmente adecuada para aplicaciones de este tipo, que combinan una gran cantidad de datos externos con la necesidad de interactividad en el cliente. La separación entre la obtención de datos (servidor) y la presentación interactiva (cliente) simplifica el flujo de datos y reduce las fuentes de error, de esta forma es posible obtener los datos en paralelo de múltiples APIs en el servidor sin coste para el cliente, lo que permite reducir de forma significativa el tiempo de carga inicial frente a enfoques donde casi todo el procesamiento se realiza en el navegador.

En segundo lugar, la integración de OAuth 2.0 mediante API Routes de Next.js ha demostrado ser una estrategia robusta y segura sin necesidad de librerías de autenticación genéricas. La decisión de almacenar los tokens en cookies \texttt{httpOnly} y resolver la autenticación en servidor sigue las buenas prácticas descritas en OAuth 2.0 (RFC 6749) \parencite{Hardt:2012} y en las recomendaciones actuales de OWASP para este tipo de flujos \parencite{OWASP:2024}. Además, la implementación manual permitió gestionar con mayor precisión tanto la renovación de tokens como los errores específicos de la API de Trakt.tv.

En tercer lugar, la estrategia de caché diferenciada por tipo de dato ha demostrado ser eficaz en todas sus capas: ISR en servidor para el catálogo, \texttt{no-store} para los datos de usuario protegidos por autenticación y una caché en cliente de cuatro niveles (memoria del proceso, sessionStorage, localStorage y deduplicación en vuelo) para las puntuaciones externas de IMDb, Trakt y OMDb en las vistas de listas, reduciendo drásticamente el número de llamadas a APIs externas.

En cuarto lugar, la integración con Plex Media Server confirma que es viable unificar en una misma aplicación el contenido local del usuario y la información procedente de servicios en la nube. Para ello, se implementó un flujo de autenticación basado en JWT con renovación automática de tokens y un mecanismo de fallback entre HTTP y HTTPS, lo que reduce fallos de conexión en entornos domésticos con distintas configuraciones de red.

Finalmente, en el plano de diseño, la combinación de animaciones con Framer Motion, superficies semitransparentes con desenfoque de fondo (\textit{glassmorphism}) y una navegación adaptativa según el dispositivo (barra horizontal en escritorio y \textit{drawer} con \textit{tab bar} en móvil) permitió construir una interfaz coherente, reconocible y más cercana a patrones de uso actuales. Estas decisiones no solo aportan diferenciación visual frente a otras plataformas de \textit{tracking}, sino que también mejoran la legibilidad, la jerarquía de la información y la fluidez de interacción entre vistas.

\section{Discusión}

El trabajo presenta algunas limitaciones que deben reconocerse.

\textbf{Ausencia de tests automatizados.} El proyecto no incluyó una suite de tests unitarios ni de integración durante el desarrollo. La validación se realizó exclusivamente mediante pruebas manuales. Los componentes con lógica de negocio compleja (cálculo de progreso de episodios, normalización de fechas para Trakt, estrategia de selección de \textit{backdrops}, algoritmo de deduplicación en vuelo) serían candidatos prioritarios para la cobertura de tests unitarios y la integración de Jest y React Testing Library sería el siguiente paso natural del proyecto.

\textbf{Dependencia de APIs de terceros.} La plataforma depende de la continuidad y estabilidad de cinco APIs externas (TMDb, Trakt.tv, OMDb, JustWatch y Plex), ninguna de las cuales está bajo el control del proyecto. Un cambio en los términos de servicio, límites de uso o estructura de datos de cualquiera de estas APIs podría afectar a la funcionalidad de la aplicación, por lo que es una limitación estructural de la arquitectura elegida, aceptada conscientemente por las ventajas que ofrece frente a la alternativa de construir una base de datos propia. El patrón de \textit{graceful degradation} implementado mitiga parcialmente este riesgo.

\textbf{Escalabilidad del sistema de caché.} El uso de \texttt{localStorage} para puntuaciones tiene un límite práctico de ~5--10\,MB según el navegador y este límite podría alcanzarse para listas con un gran número de elementos. La implementación del algoritmo LRU mitiga el problema pero no lo elimina completamente. Por esta razón, una migración a IndexedDB (límite ~50\,GB) sería recomendable para una mejor escalabilidad a largo plazo.

\section{Análisis de riesgos y estrategias de mitigación}

El estado actual de \textit{The Show Verse} refleja un producto funcional y desplegado, pero con una exposición natural a riesgos técnicos, legales y de adopción. El enfoque seguido ha sido integrar la mitigación en la propia arquitectura y dejar identificadas las acciones pendientes para la siguiente fase.

\subsection{Riesgos técnicos}

El principal punto de exposición técnica es la dependencia de servicios externos (TMDb, Trakt.tv, OMDb, JustWatch y Plex). Para reducir el impacto de cambios o caídas parciales, la aplicación encapsula las integraciones en módulos y rutas API propias, de modo que una variación en un proveedor no obliga a modificar toda la interfaz. Además, la estrategia de caché multinivel y el patrón de \textit{graceful degradation} ya implementados permiten mantener una experiencia usable cuando alguna fuente responde de forma lenta o incompleta.

También existe riesgo operativo en los flujos autenticados de Trakt, especialmente cuando un token expira o el usuario revoca permisos. Este escenario se mitiga con renovación de token desde el servidor y manejo explícito de errores de autorización, evitando estados incorrectos en el cliente y permitiendo la reconexión controlada.

En cuanto al rendimiento bajo carga, el proyecto parte con ventaja por el uso de Server Components, fetching paralelo e ISR en catálogo, que reducen llamadas repetidas. Como línea de refuerzo para crecimiento de usuarios, queda planificado endurecer políticas de reintento y observabilidad para anticipar límites de consumo antes de que afecten al servicio.

\subsection{Riesgos legales y de privacidad}

La aplicación trabaja con datos y recursos sujetos a condiciones de uso de terceros, por lo que la continuidad del proyecto depende de mantener conformidad con sus términos. En la práctica, esto implica conservar atribuciones visibles, evitar usos no permitidos de metadatos e imágenes, y revisar periódicamente cambios en políticas de API para adaptar el comportamiento cuando sea necesario.

Desde el punto de vista de protección de datos, el proyecto ya aplica una medida clave: los tokens se gestionan en cookies \texttt{httpOnly} y la autenticación se resuelve en servidor, minimizando exposición en cliente. No obstante, para una explotación pública más amplia sigue siendo necesario completar el marco documental de privacidad (información al usuario, gestión de consentimiento y procedimientos de eliminación/exportación de datos) para asegurar cumplimiento sostenido en contexto RGPD.

\subsection{Riesgos de producto y adopción}

En el plano de mercado, el riesgo no está en la viabilidad técnica inmediata, sino en la capacidad de diferenciarse de alternativas ya consolidadas. La mitigación realista consiste en reforzar aquello que ya aporta valor diferencial en el estado actual: integración transversal de fuentes, control de visionado por episodios, biblioteca unificada con Plex y una presentación visual más cuidada que la de herramientas tradicionales.

Existe además un riesgo de abandono temprano si el usuario no percibe rápido esa propuesta de valor. Por ello, las acciones más efectivas a corto plazo son mejorar onboarding y comunicación de funcionalidades clave, además de mantener una evolución incremental centrada en utilidades concretas antes que en expansiones de gran alcance.

\subsection{Síntesis operativa}

Con la arquitectura actual, el riesgo residual es asumible para una fase MVP avanzada: la base técnica es sólida, las mitigaciones críticas ya están parcialmente incorporadas en diseño y queda como prioridad cerrar el bloque de cumplimiento legal y observabilidad operativa para escalar con mayor garantías.

A partir del trabajo realizado, se identifican las siguientes líneas de desarrollo futuro:

\textbf{Implementación de Progressive Web App (PWA).} La conversión de la aplicación en una PWA, mediante la adición de un \textit{Service Worker} y un \textit{manifest} web, permitiría la instalación en dispositivos móviles y habilitaría un modo de funcionamiento \textit{offline} básico. El \textit{Service Worker} podría cachear las imágenes de pósters y los datos de catálogo para consulta sin conexión. Next.js 13+ soporta PWA mediante el plugin \texttt{next-pwa}.

\textbf{Sistema de recomendaciones personalizado.} El historial de visionado acumulado por el usuario constituye un conjunto de datos con potencial para la implementación de algoritmos de recomendación colaborativa o basada en contenido. TensorFlow.js en el cliente o una función \textit{serverless} con Python + scikit-learn permitirían generar recomendaciones personalizadas complementarias a las de Trakt. La técnica de \textit{matrix factorization} (SVD) o redes neuronales de \textit{collaborative filtering} serían candidatas naturales.

\textbf{Expansión de las funcionalidades sociales.} La comparación del historial entre usuarios, las recomendaciones de amigos y la visualización del perfil público de otros usuarios de Trakt son funcionalidades que la API de Trakt soporta pero no han sido implementadas en la versión actual. Un módulo de «Amigos» con feed de actividad reciente aumentaría significativamente el componente social de la plataforma.

\textbf{Backend propio y base de datos local.} A largo plazo, la eliminación de la dependencia total de Trakt.tv como fuente de verdad requeriría la implementación de un \textit{backend} propio con base de datos (PostgreSQL o MySQL) que almacenara el historial y las preferencias localmente. Esto permitiría además implementar funcionalidades propias como sesiones de visionado grupal (\textit{watch parties}), sistemas de \textit{achievements} y estadísticas avanzadas no disponibles en Trakt.

\textbf{Integración con más plataformas.} La arquitectura modular del sistema de integración API permitiría añadir fácilmente nuevas fuentes: MyAnimeList para anime, Goodreads para literatura adaptada, Metacritic para puntuaciones adicionales, o incluso Spotify/Apple Music para bandas sonoras. Cada integración seguiría el mismo patrón: API Route de \textit{proxy}, caché multinivel, \textit{graceful degradation}.

\textbf{Sistema de notificaciones push.} Notificaciones web (\textit{Web Push API}) para alertar al usuario de:
\begin{itemize}
\item Nuevos episodios de series seguidas.
\item Respuestas a comentarios en Trakt.
\item Nuevas películas añadidas al catálogo que coinciden con sus gustos.
\item Amigos que han visto contenido que el usuario tiene en pendientes.
\end{itemize}

\textbf{Análisis avanzado con IA.} Integración con modelos de lenguaje (GPT-4, Claude) para:
\begin{itemize}
\item Generación de resúmenes personalizados de temporadas.
\item Respuestas a preguntas sobre la trama sin spoilers.
\item Recomendaciones explicadas («Te recomiendo X porque...»).
\item Detección de contenido similar por temas y no solo por género.
\end{itemize}
