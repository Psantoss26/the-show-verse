Resumen




Resumen

El presente Trabajo de Fin de Grado aborda el diseño y la implementación de The Show Verse,
una aplicación web avanzada de gestión y descubrimiento de películas y series que permite a los
usuarios unificar todo el contenido audiovisual en un solo lugar. La plataforma cuenta con secciones
personalizadas para cada usuario, gestionadas a través de APIs externas, e integra funcionalidades
de biblioteca local mediante Plex Media Server.
La dispersión del contenido entre las distintas plataformas de streaming genera una experiencia de
usuario fragmentada y dificulta el seguimiento del contenido consumido. The Show Verse soluciona
este problema ofreciendo una plataforma que unifica el contenido de las principales páginas de
puntuaciones y plataformas de streaming haciendo uso de APIs, además de integrar el contenido
local del usuario.
El trabajo consiste en el desarrollo completo de una aplicación web en Next.js que integra las
APIs de The Movie Database (TMDb), Trakt.tv, OMDb, JustWatch y Plex Media Server. Se ha
implementado un flujo de autenticación OAuth 2.0 completo con Trakt.tv y TMDb para permitir
la sincronización del historial, los favoritos, la lista de pendientes, el calendario de estrenos y las
valoraciones del usuario. Además, se ha diseñado un sistema de caché en múltiples niveles (ISR,
memoria de proceso, sessionStorage, localStorage) con deduplicación en vuelo que reduce en un
70–80 % el número de peticiones a APIs externas en las vistas de listas.
La arquitectura implementada combina Server Components de React para la obtención inicial de
datos con Client Components para la interactividad, logrando tiempos de carga (LCP) de 1,8 s y una
puntuación de Performance en Lighthouse de 92/100. Se ha desarrollado un middleware Edge para
optimización SEO con detección de bots y generación de metadatos Open Graph, un sistema de
navegación responsive adaptativo (barra horizontal en desktop, drawer lateral + tab bar en móvil), y
componentes con efectos visuales avanzados mediante Canvas 2D y Framer Motion.
Entre las funcionalidades destacables se incluyen: seguimiento episódico con visualización de
progreso por temporada, grid de ratings que permite detectar patrones de calidad, integración con
comentarios y listas de comunidad Trakt, sistema de artwork personalizado con sobrescrituras
manuales, tres vistas alternativas (Grid, List, Compact) con persistencia de preferencias,
actualización optimista de estado en todas las acciones de usuario, y gestión de errores con graceful

Abstract




Abstract

This Final Degree Project presents the design and implementation of The Show Verse, a web
application for the management and discovery of films and TV series that allows users to unify all
audiovisual content in a single platform. The application integrates the TMDb, Trakt.tv and OMDb
APIs for metadata and personal tracking, JustWatch for streaming availability, and Plex for access
to content on a local media server. A complete OAuth 2.0 authentication flow with Trakt.tv has
been implemented, together with a multi-level cache strategy that reduces external API calls by up

modes (Grid, List and Compact), and Lighthouse scores of 92/100 in performance and 100/100 in
SEO.

1.1 Justificación y contexto

En los últimos años, el consumo de películas y series ha cambiado radicalmente debido a que la
mayoría del contenido se distribuye a través de plataformas de streaming como Netflix, HBO Max,
Disney+, Amazon Prime Video o Movistar+, cada una con su propio catálogo independiente, lo que
genera un problema habitual para cualquier usuario porque el contenido está disperso en múltiples
servicios y no existe ninguna herramienta que permita gestionarlo todo desde un único lugar.

Esto provoca situaciones como perder el seguimiento de una serie que se dejó a medias, no recordar
qué episodios se han visto, tener que buscar en varios sitios a la vez si una película está disponible
en streaming, consultar las puntuaciones y críticas de otros usuarios, o simplemente no tener un
registro de lo que ya se ha visto. Las plataformas de tracking que existen actualmente, como Trakt.tv
o Letterboxd, resuelven parte del problema pero presentan sus propias limitaciones con falta de
funcionalidades.

The Show Verse nace como respuesta a esa necesidad ofreciendo una plataforma web que centraliza
gran parte de la gestión del contenido audiovisual, integrando los datos de múltiples fuentes
mediante APIs externas y ofreciendo al usuario una experiencia completa y funcional.

Desde el punto de vista técnico, el proyecto aborda varios retos propios del desarrollo web actual
como la integración de APIs REST de terceros con distintos esquemas de autenticación y formatos
de datos, la implementación de la autenticación para el acceso con TMDb y Trakt.tv, el diseño de
una estrategia de caché en varios niveles para optimizar el rendimiento, y la construcción de una
interfaz responsiva con un diseño propio y cómodo para el usuario.


1.1.1 Motivación

El interés por el desarrollo de este proyecto surge de la experiencia personal como consumidor
habitual de películas y series a través de diversas plataformas de streaming. La dispersión del

1.2 Planteamiento del problema

El problema abordado en este trabajo puede descomponerse en varias categorías técnicas.
1. Integración de fuentes de datos. Los datos necesarios para una plataforma de gestión de

claves privadas de la aplicación.
3. Rendimiento y experiencia de usuario. Este tipo de aplicación maneja grandes volúmenes de
imágenes de alta resolución y realiza múltiples llamadas a APIs externas con límites de peticiones
en algunos casos reducidos. Es importante que haya un equilibrio entre datos almacenados en caché
y aquellos que se muestran en tiempo real.
4. Sincronización de estado entre servidor y cliente. Las acciones del usuario (marcar como
visto, añadir a favoritos, puntuar) deben reflejarse de forma inmediata en la interfaz sin esperar

1.4 Modelo de negocio

The Show Verse se ha desarrollado como proyecto académico, pero su arquitectura permite
analizar su viabilidad en un escenario real. No obstante, al tratarse de una aplicación fuertemente
dependiente de APIs de terceros, la sostenibilidad económica no puede evaluarse solo por costes
de infraestructura si no que también está condicionada por licencias de uso, límites de petición y
cambios de política de proveedores externos.


1.4.1 Viabilidad real y dependencia de APIs

Desde una perspectiva técnica y de producto, el proyecto es viable como servicio de nicho orientado
a usuarios avanzados que desean centralizar seguimiento, valoraciones y descubrimiento en una
única interfaz. Sin embargo, su potencial de monetización está limitado por la naturaleza de los
datos consumidos: una parte relevante del valor funcional procede de metadatos y catálogos de
plataformas externas cuya explotación comercial no depende exclusivamente del desarrollador.

En consecuencia, un despliegue público a pequeña escala es factible, pero una monetización agresiva
(suscripción, publicidad intensiva o reventa de datos agregados) requeriría revisar de forma previa
la conformidad legal y contractual con cada proveedor.

      condicionada por límites de uso y por características del ecosistema Trakt.

1.4.4 Conclusión de viabilidad

La viabilidad del proyecto es media para un producto independiente y alta como plataforma

gobernanza de dependencias externas: condiciones de uso, posibles acuerdos comerciales y cambios
de política de APIs.
Por tanto, antes de cualquier lanzamiento comercial, el paso crítico no sería añadir más
funcionalidades, sino formalizar un marco de cumplimiento con proveedores y revisar
periódicamente costes y límites vigentes, ya que estos parámetros pueden variar en el tiempo.

2.1 Marco teórico del trabajo

2.1.1 Las plataformas de streaming y sus consecuencias para el usuario

La transición del modelo tradicional al streaming bajo demanda se inició con el lanzamiento del
servicio de Netflix en 2007, aunque su consolidación definitiva se produjo a partir de 2019, con la
irrupción simultánea de Disney+, Apple TV+ y HBO Max. En 2024, el usuario medio en mercados
como España o Estados Unidos tiene acceso simultáneo a entre 3 y 5 servicios de streaming distintos,
según datos de CNMC y Statista (CNMC, 2024; Statista, 2024).

Esta fragmentación genera una sobrecarga de elección en cada usuario, lo que provoca que inviertan
una gran cantidad de tiempo en decidir qué ver, olviden mantener un registro de lo visto y pierdan
el seguimiento de las series que consumen de forma no continua (Schwartz, 2004).

La necesidad de una capa de abstracción sobre las plataformas de streaming ha generado el
desarrollo de aplicaciones conocidas como media trackers.


2.1.2 Protocolos de autenticación: OAuth 2.0

OAuth 2.0 es el estándar que permite a una aplicación actuar en nombre de un usuario en un servicio
externo sin que ese servicio tenga que proporcionarle su contraseña (Hardt, 2012). En The Show
Verse se utiliza para conectar la cuenta de Trakt.tv del usuario con la aplicación.

El flujo implementado (Authorization Code) funciona de la siguiente manera:



                                                  15
Capítulo 2. Estado de la cuestión


1) Cuando el usuario pulsa «Conectar con Trakt», la ruta /api/trakt/auth/start genera
   un código aleatorio de seguridad (state) para evitar ataques CSRF, construye la URL de
   autorización de Trakt y redirige al usuario.

2) El usuario introduce sus credenciales en Trakt y concede permisos. Trakt redirige de vuelta a
   /api/trakt/auth/callback incluyendo un code de un solo uso y el mismo state.

3) La ruta de callback verifica el state e intercambia el code por un access_token y un
   refresh_token directamente desde el servidor.

4) Ambos tokens se guardan en cookies httpOnly con los atributos Secure y
   SameSite=Lax, de forma que el JavaScript del navegador no puede leerlos en ningún
   momento.

5) Cuando el access_token expira, la aplicación lo renueva automáticamente usando el
   refresh_token.


2.1.3 APIs REST

Las APIs utilizadas en este proyecto siguen el estilo REST (Fielding, 2000) con las siguientes
especificidades:

TMDb API v3 es una API REST pública con autenticación por API Key. Proporciona acceso a su
base de datos de más de 800.000 películas y 150.000 series de televisión, con soporte para más de
40 idiomas. Su límite de peticiones es de 40 peticiones por 10 segundos para cuentas gratuitas (The
Movie Database, 2024).

Trakt.tv API v2 combina autenticación por API Key para endpoints públicos con OAuth 2.0 Bearer
Token para endpoints autenticados. Un aspecto técnico relevante es el sistema de identificadores:
Trakt mantiene sus propios slugs e IDs, pero acepta identificadores de IMDb y TMDb mediante el

2.1.5 Diseño de interfaz

El diseño de The Show Verse se basa en el uso de los backdrops y posters obtenidos de TMDb como

Modo oscuro y backdrops de fondo. El color de base de la aplicación es el negro (#000000).

2.2 Trabajos relacionados

Letterboxd (Letterboxd, 2024) (fundado en 2011, Auckland, Nueva Zelanda) es la plataforma de
referencia para el registro y la crítica cinematográfica personal. Con más de 20 millones de usuarios
registrados en 2024, su principal fortaleza es la componente social y el sistema de reseñas. Su
limitación más significativa para los propósitos de este trabajo es la ausencia de soporte para el
seguimiento de series de televisión por episodio.

Trakt.tv (fundado en 2010) es la plataforma más completa en cuanto a funcionalidades de tracking,
con soporte completo para películas y series a nivel de episodio. Su característica diferencial es
la API pública v2, documentada y con límites de uso generosos para desarrolladores registrados.
Sin embargo, su interfaz de usuario presenta un diseño anticuado que no ha evolucionado
significativamente en la última década.

Simkl (fundado en 2012) ofrece funcionalidades similares a Trakt con una interfaz algo
más moderna, pero con un ecosistema de API menos estable y una comunidad de usuarios
significativamente menor.

JustWatch (fundado en 2014) se especializa en la localización de contenido entre plataformas de
streaming, pero carece de funcionalidades de tracking personal.

La comparación entre estas plataformas pone de manifiesto una oportunidad en el mercado: ninguna
combina la funcionalidad de Trakt (seguimiento por episodio, sincronización, valoraciones) con una
interfaz de usuario moderna y responsiva comparable a las propias plataformas de streaming. The
Show Verse se posiciona precisamente en este punto.

Capítulo 3
Aspectos metodológicos

El desarrollo de The Show Verse se ha abordado mediante una metodología iterativa basada en
el ciclo de vida ágil, adaptada a un proyecto individual. La elección de este enfoque frente a las
metodologías en cascada se justifica por la naturaleza exploratoria del proyecto: la integración con
APIs de terceros implica incertidumbres técnicas que no siempre se pueden anticipar en una fase
de análisis, y la prioridad entre funcionalidades puede variar a medida que se comprende mejor el
problema.
El proceso de desarrollo se articula en iteraciones de aproximadamente una semana de duración. Al
inicio de cada iteración se seleccionan del backlog las funcionalidades a abordar, priorizadas según
tres criterios:

  •   Dependencia técnica: las funcionalidades que sirven de base a otras (autenticación, cliente de
      API) se abordan primero.

  •   Valor aportado al usuario: se priorizan las funcionalidades más visibles y relevantes para la
      experiencia de uso.

  •   Riesgo técnico: las integraciones más inciertas se adelantan para conocer cuanto antes sus
      restricciones.


3.1 Metodología

3.1.1 Criterios de integración de APIs

La selección de las APIs externas ha seguido los siguientes criterios:

  •   Datos completos: TMDb fue seleccionada frente a otras alternativas (The Open Movie
      Database, Wikidata) por ofrecer la mayor cobertura de contenido, soporte en diferentes idiomas,
      galería de imágenes y datos de streaming, además de disponer de la documentación más
      completa y una comunidad de desarrolladores activa.

                                                  18
Capítulo 3. Aspectos metodológicos

 •   Capacidad de tracking: Trakt.tv fue seleccionada como plataforma de sincronización por su
     API pública v2 bien documentada, su soporte completo de series a nivel de episodio y por ser la
     plataforma de tracking más referenciada por la comunidad de desarrolladores independientes.

 •   Complementariedad: OMDb fue seleccionada únicamente para los datos de rating de IMDb y
     Rotten Tomatoes, que TMDb no proporciona directamente.


3.1.2 Criterios de decisión para la autenticación

El problema de la autenticación OAuth 2.0 admitía varias opciones de implementación en Next.js. Se
descartó el uso de librerías de autenticación genéricas como NextAuth.js / Auth.js por las siguientes
razones:

 •   Trakt.tv no es un proveedor OAuth soportado de forma nativa por estas librerías.

 •   La personalización necesaria (refresco de tokens específico de Trakt, intercambio de códigos
     con headers personalizados) habría requerido más configuración que implementar el flujo
     directamente.

 •   La implementación propia ofrece mayor control sobre el manejo de errores y el almacenamiento
     de tokens.

Se optó por implementar el flujo Authorization Code de forma manual mediante API Routes de
Next.js, almacenando los tokens en cookies httpOnly con atributos Secure y SameSite=Lax.
Esta decisión garantiza que los tokens no sean accesibles desde JavaScript del navegador (protección
frente a XSS) y que no se envíen en peticiones cross-site no intencionadas (protección parcial frente
a CSRF).


3.1.3 Estrategia de caché

de la frecuencia de actualización del dato en origen y su importancia para la experiencia de usuario.
Esta diferenciación permite reducir el número de llamadas a APIs externas en un estimado del 70–
80 % para los datos de catálogo, mientras se garantiza la actualización inmediata de los datos de
usuario que requieren frescura absoluta.

3) sessionStorage en cliente: caché de puntuaciones que persiste durante la sesión del navegador
    (hasta cerrar pestaña), común entre páginas.

4) localStorage en cliente: caché persistente de puntuaciones con TTL de 30 días, sobrevive entre
    sesiones del navegador.


Deduplicación en vuelo: implementada en lib/api/omdb.js, si múltiples componentes
solicitan el mismo dato simultáneamente, solo se realiza una petición HTTP y todas las solicitudes
esperan el resultado compartido. Esto es crítico en listas de 100+ ítems donde varias tarjetas pueden
necesitar el mismo imdbId.

Pool de concurrencia: para respetar los rate limits de las APIs (OMDb: 1000 peticiones/día en plan
gratuito), las peticiones paralelas se limitan a 4–5 simultáneas mediante un patrón de worker pool.
Cuando hay 100 peticiones pendientes, se ejecutan en lotes de 4, esperando a que una termine antes
de lanzar la siguiente.

Algoritmo LRU para localStorage: al superar 500 entradas cacheadas, se aplica Least Recently
Used para eliminar entradas antiguas y mantener el tamaño del almacenamiento controlado
(típicamente <2 MB).

4.2 Implementación del sistema de autenticación OAuth 2.0

La implementación del flujo OAuth 2.0 con Trakt.tv es el núcleo técnico más delicado del proyecto.
El flujo se articula en cuatro API Routes de Next.js:

Inicio del flujo (/api/trakt/oauth): construye la URL de autorización de Trakt con los
parámetros client_id, redirect_uri y response_type=code, y redirige al usuario.
Este endpoint se invoca al pulsar el botón «Conectar con Trakt» de la Navbar.

Callback (/api/trakt/auth/callback): recibe el authorization_code de Trakt, lo
intercambia por access_token y refresh_token mediante una petición POST servidor-a-
servidor a https://api.trakt.tv/oauth/token, y almacena ambos tokens en cookies
httpOnly. A continuación redirige al usuario a la página de inicio.

Estado (/api/trakt/auth/status): lee las cookies de sesión y, si existe access_token
válido, hace una petición a /users/me de Trakt para obtener el perfil del usuario. Si el token
está expirado, intenta refrescarlo usando el refresh_token antes de declarar al usuario como no
autenticado.

Desconexión (/api/trakt/auth/disconnect): revoca el token en Trakt mediante la API
/oauth/revoke y elimina las cookies de sesión.

Este diseño garantiza que ni el client_secret de Trakt ni los access_token del usuario
sean accesibles desde el JavaScript del navegador en ningún momento del flujo.

4.4 Arquitectura de API Routes propias

Next.js permite definir API Routes como funciones serverless que se ejecutan en el servidor,
proporcionando endpoints HTTP personalizados. El proyecto implementa 74 API Routes
organizadas en el directorio src/app/api/, que actúan como capa intermedia entre el cliente
y las APIs externas, implementando lógica de negocio, transformación de datos, caché y seguridad.


4.4.1 Funciones principales de las API Routes

Las API Routes del proyecto cumplen seis funciones fundamentales:

1. Proxy seguro. Protegen las claves API (API keys) de exposición al navegador. Todas las
peticiones a TMDb, Trakt, OMDb, Plex y JustWatch pasan por endpoints intermedios que almacenan
las credenciales en variables de entorno del servidor, garantizando que nunca se envíen al cliente.

2. Transformación y enriquecimiento de datos. Las respuestas de las APIs externas se procesan
antes de enviarlas al cliente: se traducen identificadores entre sistemas (TMDb ID ↔ Trakt ID ↔
IMDb ID), se combinan datos de múltiples fuentes en una única respuesta, y se añaden metadatos
adicionales como títulos, posters y años de estreno desde TMDb.

3. Caché y optimización. Implementan caché en memoria para reducir llamadas redundantes a APIs
externas con límites de tasa. Las respuestas exitosas se almacenan temporalmente (de 60 segundos a
24 horas según el tipo de dato) y se comparten entre peticiones concurrentes mediante deduplicación.

4. Gestión de autenticación. Manejan flujos OAuth 2.0 completos para Trakt y TMDb,
almacenando tokens de sesión en cookies HTTP-only inaccesibles desde JavaScript. Implementan
renovación automática de tokens caducados y validación de estado de conexión.

5. Agregación de datos. Consultan múltiples APIs en paralelo y combinan los resultados.
Por ejemplo, /api/tv/[id]/ratings obtiene puntuaciones de TMDb, Trakt e IMDb
simultáneamente y las devuelve en un objeto unificado.

6. Validación y normalización. Validan parámetros de entrada, normalizan formatos (ej: ratings
de 1–10 de Trakt se redondean hacia arriba cuando provienen de TMDb que acepta decimales) y
gestionan timeouts configurables para evitar bloqueos.

4.4.2.3 Valoraciones y ratings

Gestión de ratings personales. GET/POST /api/trakt/ratings implementa CRUD

 •   POST con rating: null: Elimina la valoración existente del título, equivalente a DELETE.

 •   Implementa auto-refresh de tokens: si detecta un token caducado, ejecuta el flujo de renovación
     automáticamente y reintenta la petición original.


El endpoint paralelo POST /api/tmdb/movies/[id]/rating gestiona valoraciones en

Ratings agregados de múltiples fuentes. GET /api/tv/[id]/ratings es un endpoint de
agregación especializado para series que combina datos de tres fuentes:


1) Obtiene el rating promedio de TMDb directamente desde /tv/{id}.

2) Consulta el scoreboard de Trakt mediante /search/tmdb/{id} para obtener la valoración
     comunitaria.

3) Extrae el IMDb ID de los identificadores externos de TMDb, y consulta OMDb para obtener el

4.4.3 Patrones técnicos transversales

Todos los endpoints implementan patrones comunes de robustez:
Manejo de errores estructurado. Las respuestas de error siguen un formato consistente: {error:
“descripción”, code?: “ERR_CODE”, details?: {...}}. Los códigos HTTP se
asignan semánticamente: 400 para parámetros inválidos, 401 para autenticación requerida, 404 para
recursos no encontrados, 429 para rate limit, 500 para errores internos, 503 para APIs externas no
disponibles.
Validación de entrada. Los parámetros se validan antes de ejecutar lógica costosa: TMDb IDs
deben ser numéricos positivos, fechas deben cumplir formato ISO 8601, ratings se restringen a
rangos válidos (1–10), y se rechazan requests con tamaños excesivos (límite de 10000 IDs en batch
queries).
Timeouts y circuit breakers. Todas las peticiones a APIs externas incluyen timeouts configurables
(valores típicos: 5s para datos críticos, 10s para búsquedas, 15s para operaciones batch). Si un
servicio externo falla repetidamente, se activa degradación graceful retornando datos parciales o
cacheados.
Logging y observabilidad. Los errores se loguean con contexto completo (endpoint, parámetros,
usuario si está autenticado, timestamp, stack trace) permitiendo debugging en producción. Las
peticiones lentas (>3 segundos) se marcan como warnings para identificar cuellos de botella.

4.5.2.1 Arquitectura de datos

Se hace uso de Server Component y Client Component que optimiza tanto el SEO como la
interacción en la página web. El Server Component se encarga de obtener los datos críticos iniciales
en tiempo de renderizado server-side, incluyendo los metadatos completos del título, los logos
oficiales, el reparto y equipo técnico completo, las recomendaciones de contenido similar, los
proveedores de streaming disponibles por país, los videos oficiales (trailers, teasers, clips) y las
reseñas de la comunidad. Si el usuario está autenticado, también se obtiene el estado de la cuenta (si
ha marcado el título como favorito, si está en su lista de pendientes, y su valoración personal).

Una vez que el Client Component se ejecuta en el navegador, se empiezan a realizar peticiones
paralelas para ir construyendo la interfaz. Primero se carga el scoreboard de Trakt mediante
/api/trakt/scoreboard para obtener la puntuación, el número de usuarios que están
viéndolo actualmente (watchers), el número total de reproducciones (plays), cuántos usuarios
lo tienen en sus bibliotecas (collectors) y cuántas listas de Trakt lo incluyen. En paralelo, si el
título tiene IMDb ID, se consulta OMDb mediante /api/omdb para obtener el rating de IMDb,
el número de votos, las puntuaciones de Rotten Tomatoes y Metacritic, y los premios ganados.
Todos estos datos se guardan en caché: OMDb en sessionStorage con TTL de 24 horas, y el
scoreboard de Trakt en memoria con TTL de 5 minutos.

A partir de la API de Plex se detecta si el título está disponible en la biblioteca personal y se guarda
durante 7 días en sessionStorage y se generan múltiples URLs de reproducción: una para el
navegador web (plexUrl.web) y otra para la aplicación móvil de Plex (plexUrl.mobile).

Los proveedores de streaming se obtienen de dos fuentes distintas. Por un lado, TMDb proporciona
información básica de la disponibilidad mediante watch/providers, indicando si el título está
disponible para alquiler, compra o en plataformas de suscripción. Por otro lado, JustWatch (mediante
la API no oficial en GraphQL) proporciona enlaces directos al contenido en cada plataforma. Si
la película pertenece a una colección (por ejemplo, la saga de El Señor de los Anillos), se carga
la información completa de la colección mediante /api/tmdb/collection y se muestra una
nueva sección con un carrusel y todas las películas de la saga, ordenadas cronológicamente.

y los muestra en formato grid o lista según la preferencia del usuario (guardada en
localStorage). Cada episodio se renderiza con su número, nombre, fecha de emisión,
puntuación de TMDb, puntuación de Trakt (si está disponible), y un checkbox interactivo que
permite marcar/desmarcar el episodio como visto de forma individual. Cuando el usuario marca
un episodio como visto, se ejecuta la función toggleEpisodeWatched(seasonNumber,
episodeNumber) que realiza una actualización del checkbox y envía una petición POST
a /api/trakt/episode/watched con el payload {showId, season, episode,
watchedAt: new Date().toISOString()} y si tiene éxito, se recarga el estado completo
de la serie mediante GET /sync/watched/shows/{id} para sincronizar con cualquier
cambio que haya podido ocurrir en Trakt desde otros dispositivos o aplicaciones.
El componente EpisodeRatingsGrid.jsx ubicado en una sección de la página de detalles
ofrece otro tipo de vista de todos los episodios de la serie. Cada celda del grid representa
un episodio y tiene un color en función de la puntuación, es completamente interactivo y se
puede hacer clic en cualquier celca para acceder a la página de detalles del episodio en cuestión
(/details/tv/{showId}/season/{season}/episode/{number}).
Para optimizar el rendimiento en series con cientos de episodios, las puntuaciones se cargan
en paralelo mediante Promise.all(seasons.map(s =>getSeasonDetails(showId,
s.season_number))) en lugar de secuencialmente. Esto reduce la latencia total de
aproximadamente N segundos (donde N es el número de temporadas).

normalizado.

API interna /api/links/justwatch recibe el título y el año, genera un slug normalizado
eliminando caracteres especiales y convirtiendo espacios en guiones, construye la URL final

verificar que el recurso existe (código HTTP 200), y cachea el resultado en localStorage de
forma indefinida para evitar validaciones repetidas pero si la validación falla, devuelve una URL
de búsqueda genérica como fallback. Para Letterboxd (exclusivo de películas), si el título tiene
IMDb ID se construye directamente la URL https://letterboxd.com/imdb/{imdbId}
y en caso contrario, se genera un slug basado en el título y año, se valida su existencia
mediante petición HEAD, y se cachea el resultado. Por último, el enlace de SeriesGraph
(exclusivo de series) utiliza directamente el TMDb ID sin validación, construyendo la URL
https://seriesgraph.com/show/{tmdbId}.

4.5.2.7 Imágenes y posters

El sistema de gestión de imágenes implementa tres modos de visualización distintos y permite
a los administradores seleccionar manualmente qué versión de poster o backdrop se muestra.
Para series de televisión, el sistema carga automáticamente las imágenes de todas las temporadas
mediante fetchTVImages({showId, apiKey}) que itera por cada temporada, obtiene sus
imágenes mediante GET /tv/{id}/season/{seasonNumber}/images, y fusiona todos
los resultados en un único array eliminando duplicados por file_path.

El sistema distingue tres tipos de imágenes según su función visual. Los posters son imágenes
verticales con aspect ratio 2:3 que se muestran en el hero de la página de detalle y se optimizan

backdrops de preview son imágenes horizontales con aspect ratio 16:9, optimizadas en tamaños
w780 y w1280.

La selección automática de la mejor imagen se realiza mediante funciones. Para posters,
pickBestPosterTV(posters, languages) ordena por resolución (preferencia por
imágenes 4K y 2K) y número de votos, priorizando posters en español o inglés si
están disponibles. Para backdrops, pickBestBackdropTVNeutralFirst(backdrops,

superpuesto) para el background de móvil, y luego ordena por resolución y votos para el resto.

Los administradores (específicamente el usuario psantos26 hardcoded en el componente) tienen
acceso a una galería administrable con tres pestañas: Posters, Backdrops (para modo preview) y
Backgrounds (para fondo móvil). Cada pestaña incluye filtros de resolución (Todos, 720p, 1080p,
2K, 4K calculados a partir del width de cada imagen) y checkboxes de idioma (ES, EN). Las
imágenes se muestran en un carrusel horizontal y al hacer clic en cualquier imagen se establece
como activa, guardando la selección en localStorage con claves específicas por tipo e ID (por
ejemplo, showverse:movie:550:poster, showverse:tv:1399:backdrop).

4.5.3 Favoritos (/favorites)

La página de favoritos muestra la colección de películas y series del usuario a partir
de la API TMDb. El componente FavoritesClient.jsx obtiene los datos mediante
dos peticiones paralelas a los endpoints GET /account/{id}/favorite/movies y
GET /account/{id}/favorite/tv, cada uno en bloques de 20 elementos. La función
fetchFavoritesForUser() implementa un patrón de carga paralela: primero ejecuta
una petición inicial para obtener el número total de páginas disponibles (total_pages), y
posteriormente lanza todas las páginas restantes mediante Promise.all(), procesándolas en
lotes de 5 peticiones concurrentes para respetar los límites de la API de TMDb. Los resultados de
películas y series se combinan en un único array marcando cada elemento con media_type:
“movie” o media_type: “tv”.

El componente implementa además un sistema de caché para evitar peticiones redundantes.
Las puntuaciones de OMDb (ratings de IMDb, Rotten Tomatoes y Metacritic) se almacenan en
sessionStorage con clave omdb:{imdbId} y un TTL de 24 horas, permitiendo que los
datos persistan durante la sesión de navegación actual sin necesidad de nuevas peticiones. Las
puntuaciones de Trakt se guardan en localStorage con TTL de 24 horas y marcas de tiempo
en formato epoch, garantizando persistencia entre sesiones. La carga de puntuaciones externas no se
realiza hasta que el usuario activa el hover sobre una tarjeta, evitando decenas de peticiones durante

4.5.15 Patrones de optimización transversales

La aplicación implementa un conjunto de patrones de optimización aplicados de forma sistemática
que explican los tiempos de respuesta percibidos como inmediatos a pesar de integrar datos de seis
APIs externas distintas.

Actualización opt imista. Todas las acciones de escritura del usuario (marcar como favorito,
añadir a pendientes, valorar) actualizan la interfaz instantáneamente sin esperar la confirmación del
servidor. La petición real se envía en paralelo en segundo plano, y solo si falla se revierte el cambio
visual mostrando un mensaje de error. Este patrón elimina la latencia perceptible de 300 a 600
milisegundos que tendría una actualización sincrónica, creando la sensación de respuesta inmediata
característica de las aplicaciones nativas.

Precarga de imágenes. Las imágenes de posters y backdrops se precargan con un margen
adelantado respecto al viewport, de forma que cuando el usuario hace scroll las imágenes ya están
disponibles en la caché del navegador, evitando tiempos de carga visibles.

Control de concurrencia. Para respetar los límites de las APIs externas sin renunciar al paralelismo,
se mantiene un máximo de 4 o 5 peticiones concurrentes. Esto reduce el tiempo de carga de
colecciones paginadas de N peticiones secuenciales a aproximadamente N/4 rondas paralelas,
mejorando el rendimiento entre 4 y 5 veces respecto a la ejecución secuencial.

Cancelación de peticiones obsoletas. Cuando el usuario navega a otra página mientras hay
peticiones en curso, estas se cancelan automáticamente para prevenir condiciones de carrera y
actualizaciones de estado en componentes ya desmontados.

Regeneración estática incremental (ISR). Los datos semi-estables como catálogos de tendencias
y listas de mejor valoradas se sirven con regeneración cada 10 a 30 minutos, permitiendo que la
mayoría de visitas utilicen páginas preconstruidas desde la CDN sin ejecutar peticiones a APIs
externas.

4.8 Sistema de caché multinivel

El rendimiento de The Show Verse depende críticamente de la estrategia de caché implementada,

4.8.1 Nivel 1: Incremental Static Regeneration (ISR)

Las páginas del catálogo (trending, top rated, populares) utilizan ISR con revalidate variable

    •   Catálogo general (10 minutos): páginas de trending y populares se regeneran cada 600 s.

    •   Temporal crítico (no-store): datos personales (favoritos, watchlist, historial) nunca se cachean
        en servidor, requieren autenticación fresca.

El módulo lib/api/tmdb.js aplica next: {revalidate: TTL} condicionalmente según

4.8.4 Nivel 3: sessionStorage (cliente)

Los componentes que muestran listas de contenido implementan caché de puntuaciones en el

Esta caché persiste durante toda la navegación del usuario, incluyendo recargas de página, pero
se elimina automáticamente al cerrar la pestaña del navegador, equilibrando utilidad y control del
almacenamiento.


4.8.5 Nivel 4: localStorage (cliente)

Para las listas personales (favoritos, pendientes), las puntuaciones se almacenan en
localStorage con un tiempo de vida de 30 días. A diferencia de sessionStorage, estos
datos persisten entre sesiones del navegador, reduciendo drásticamente el número de peticiones en
visitas recurrentes.

El sistema implementa un mecanismo de limpieza automática: cuando se superan 500 entradas
cacheadas, se aplica un algoritmo LRU (Least Recently Used) que elimina las entradas más antiguas,
garantizando que el tamaño del almacenamiento no crezca indefinidamente.

4.8.6 Deduplicación de peticiones concurrentes

Cuando múltiples componentes solicitan simultáneamente los mismos datos (por ejemplo, 10

sola solicitud real a la API externa. El resultado se comparte con todos los solicitantes, evitando
peticiones redundantes y respetando los límites de las APIs.
Esta estrategia reduce el tráfico hasta 10 veces durante la carga inicial de listas extensas, mejorando
significativamente los tiempos de respuesta.

4.9 Middleware y optimización SEO

El sistema implementa una estrategia de optimización para asegurar que los bots de redes sociales
y motores de búsqueda puedan indexar correctamente el contenido de la aplicación. Esto es
fundamental porque, al ser una aplicación React basada en componentes del lado del cliente, los
bots no ejecutarían el JavaScript y recibirían un HTML vacío sin metadatos útiles.


4.9.1 Detección de bots y reescritura de rutas

El middleware intercepta las peticiones entrantes e identifica si provienen de bots (WhatsApp,
Facebook, Twitter, Telegram, Discord, LinkedIn, Googlebot, Bingbot, etc.) mediante el análisis del
encabezado User-Agent. Cuando se detecta un bot accediendo a una página de detalle (como
/details/movie/550), el sistema reescribe transparentemente la URL a una ruta alternativa
(/s/movie/550) que sirve contenido HTML estático con todos los metadatos necesarios.
Esta reescritura es completamente transparente: la URL visible no cambia, pero internamente el bot
recibe una respuesta HTML completa con toda la información de Open Graph y Twitter Cards. El
patrón limita su ejecución exclusivamente a las rutas de detalle, evitando cualquier sobrecarga en el
resto de la aplicación.

4.9.2 Generación de HTML optimizado para compartir

Las rutas alternativas (/s/*) construyen páginas HTML estáticas que incluyen:

    •   Metadatos Open Graph: título, descripción, tipo de contenido (película, serie o persona), URL
        canónica e imagen representativa.

    •   Twitter Cards: información adaptada para previsualizaciones en Twitter/X.

    •   Selección inteligente de imágenes: se prioriza el uso de backdrops horizontales (1280×720
        píxeles) sobre posters verticales, ya que las redes sociales muestran previsualizaciones en

    •   Redirección automática para usuarios reales: si un humano accede directamente a estas

CDN sin ejecutar código en el servidor, reduciendo drásticamente la carga y mejorando los tiempos
de respuesta.

4.9.4 Resumen de la arquitectura dual de rutas

El flujo completo de una petición a /details/movie/550 funciona de la siguiente manera:


1) La petición llega al middleware Edge de Next.js.

2) El middleware evalúa el User-Agent contra la expresión regular BOT_UA.

3) Si es un bot: reescritura transparente a /s/movie/550. El Route Handler devuelve HTML
     con Open Graph completo, cacheado 24h en CDN.

4) Si es un usuario: la petición pasa sin modificar a /details/movie/550. Se ejecuta el
     Server Component que hidrata el Client Component interactivo.


Esta arquitectura garantiza:

 •   Los bots indexan las rutas /s/* con metadatos completos (OGP, Twitter Cards, canonical).

 •   Los usuarios navegan las rutas /details/* con el Client Component completo e interactivo.

 •   No hay contenido duplicado: un usuario que acceda directamente a /s/movie/550 es
     redirigido por el <meta refresh> a la ruta interactiva.

 •   El 99 % del tráfico de bots se resuelve desde la CDN sin invocar funciones serverless.

Capítulo 5

Conclusiones

El trabajo ha demostrado que los objetivos planteados son alcanzables con las tecnologías actuales
de desarrollo web y sin necesidad de infraestructura de servidor propia. Los resultados obtenidos
permiten extraer las siguientes conclusiones.

En primer lugar, la arquitectura Next.js App Router con React Server Components ha demostrado ser
particularmente adecuada para aplicaciones de este tipo, que combinan una gran cantidad de datos
externos con la necesidad de interactividad en cliente. La separación entre la obtención de datos
(servidor) y la presentación interactiva (cliente) simplifica el flujo de datos y reduce las fuentes de
error. Los Server Components permiten realizar fetching paralelo de múltiples APIs en el servidor
sin coste para el cliente, lo que resulta en tiempos de carga significativamente inferiores a los de una
SPA equivalente.

En segundo lugar, la integración de OAuth 2.0 mediante API Routes de Next.js ha probado ser
una estrategia robusta y segura sin necesidad de librerías de autenticación genéricas. El patrón de
almacenar tokens en cookies httpOnly y delegar toda la lógica de autenticación al servidor cumple
con las recomendaciones de seguridad del RFC 6749 (Hardt, 2012) y las guías de OWASP (OWASP,
2024) para aplicaciones OAuth. La implementación manual permitió además un control fino sobre
el flujo de refresco de tokens y la gestión de errores específicos de Trakt.tv.

En tercer lugar, la estrategia de caché diferenciada por tipo de dato ha demostrado ser eficaz en
todas sus capas: ISR en servidor para el catálogo, no-store para los datos de usuario protegidos
por autenticación, y una caché en cliente de cuatro niveles (memoria del proceso, sessionStorage,
localStorage y deduplicación en vuelo) para las puntuaciones externas de IMDb, Trakt y OMDb en
las vistas de listas, reduciendo drásticamente el número de llamadas a APIs externas. Las métricas
en producción muestran una reducción del 70–80 % en el tráfico API para usuarios recurrentes.

En cuarto lugar, el sistema de integración con Plex Media Server demuestra que es técnicamente
viable conectar contenido local del usuario con datos de plataformas en la nube. La implementación
del flujo de autenticación JWT con refresco automático de tokens y fallback entre HTTP/HTTPS
permite una experiencia fluida incluso en configuraciones de red complejas. Esta funcionalidad

diferencia sustancialmente el proyecto de las plataformas de tracking existentes.

En quinto lugar, la implementación del middleware Edge para detección de bots y reescritura de
URLs ha demostrado ser una solución eficiente para el problema de SEO en aplicaciones con Client
Components. El patrón de rutas duales (/details/ para usuarios, /s/ para bots) con redirección
automática garantiza metadatos Open Graph completos sin duplicar lógica de negocio.

Finalmente, desde el punto de vista del diseño, el uso de Framer Motion para las animaciones, el
sistema de superficies semitransparentes con desenfoque de fondo (glassmorphism) y la arquitectura
de navegación adaptativa (barra horizontal en desktop, drawer + tab bar en móvil) ha producido una

5.2 Discusión

El trabajo presenta algunas limitaciones que deben reconocerse.

Ausencia de tests automatizados. El proyecto no incluyó una suite de tests unitarios ni de
integración durante el desarrollo. La validación se realizó exclusivamente mediante pruebas
manuales. Los componentes con lógica de negocio compleja (cálculo de progreso de episodios,
normalización de fechas para Trakt, estrategia de selección de backdrops, algoritmo de
deduplicación en vuelo) serían candidatos prioritarios para la cobertura de tests unitarios. La
integración de Jest y React Testing Library sería el siguiente paso natural del proyecto.

Dependencia de APIs de terceros. La plataforma depende de la continuidad y estabilidad de cinco
APIs externas (TMDb, Trakt.tv, OMDb, JustWatch y Plex), ninguna de las cuales está bajo el
control del proyecto. Un cambio en los términos de servicio, límites de uso o estructura de datos
de cualquiera de estas APIs podría afectar la funcionalidad de la aplicación. Esta es una limitación
estructural de la arquitectura elegida, aceptada conscientemente por las ventajas que ofrece frente a
la alternativa de construir una base de datos propia. El patrón de graceful degradation implementado
mitiga parcialmente este riesgo.

Accesibilidad parcial. La puntuación de accesibilidad en Lighthouse (88/100) indica que, aunque se
han aplicado prácticas básicas (contraste de color, etiquetas ARIA, textos alternativos en imágenes),


                                                  85
Capítulo 5. Conclusiones


no se ha alcanzado el nivel WCAG 2.1 AA completo (W3C, 2023). En particular, la navegación
exclusiva por teclado en los modales de episodios y el sistema de tres vistas requiere trabajo
adicional. La implementación de react-focus-lock y la auditoría con lectores de pantalla
(NVDA, VoiceOver) serían pasos necesarios para alcanzar conformidad total.

Escalabilidad del sistema de caché. El uso de localStorage para puntuaciones tiene un límite
práctico de 5–10 MB según navegador. Para usuarios con listas de 500+ favoritos, este límite
podría alcanzarse. La implementación del algoritmo LRU mitiga el problema pero no lo elimina
completamente. Una migración a IndexedDB (límite 50 GB) sería recomendable para escalabilidad
a largo plazo.

Integración de JustWatch no oficial. La ausencia de API oficial de JustWatch obligó a utilizar
los endpoints GraphQL internos de su aplicación web. Aunque funcionan de forma estable, no hay
garantía de continuidad. Cualquier cambio en la estructura de las queries GraphQL requeriría adaptar
el cliente.


5.3 Análisis de riesgos y estrategias de mitigación

La viabilidad de The Show Verse depende de factores técnicos, legales y comerciales que presentan
riesgos inherentes. Se ha identificado y documentado cada riesgo significativo junto con estrategias
de mitigación.


5.3.1 Riesgos técnicos

Cambios incompatibles en APIs externas. Riesgo: Las cinco APIs en las que se basa la aplicación
(TMDb, Trakt.tv, OMDb, JustWatch, Plex) podrían cambiar su estructura de respuesta, parámetros o
métodos de autenticación, generando errores en producción. Severidad: Alta. Probabilidad: Media
(TMDb y Trakt tienen historial de mantenimiento de compatibilidad hacia atrás; JustWatch y Plex
son menos previsibles). Mitigación:

  •   Implementar versionado explícito en llamadas a APIs (ej. ‘https://api.themoviedb.org/3/...‘
      incluye versión en URL).

  •   Mantener un sistema de alertas para cambios en documentación de APIs.

  •   Implementar graceful degradation: si una API falla, mostrar versión en caché anterior en lugar
      de error.

  •   Desarrollar cliente de fallback para APIs críticas (ej. TheMovieDB como fallback si TMDb no
      responde).

                                                 86
Capítulo 5. Conclusiones


Fragmentación de permisos OAuth en Trakt. Riesgo: El usuario podría revocar permisos en
Trakt sin avisar a la aplicación, causando fallos silenciosos en operaciones que requieren acceso.
Severidad: Media. Probabilidad: Baja (evento raro pero posible). Mitigación:

 •   Verificar el estado de permiso desde RefreshTokenStrategy cada 7 días.

 •   Capturar errores 401 Unauthorized y prompts de reconexión clara.

 •   Almacenar última validación de permisos en cookies con TTL.

Límites de rate limit en APIs. Riesgo: TMDb limita a 40 req/10s para cuentas gratuitas. Un pico
de uso podría exceder este límite durante horarios de máxima audiencia global. Severidad: Media.
Probabilidad: Baja con el tamaño actual de usuarios. Mitigación:

 •   Implementar backoff exponencial: si rate limit se alcanza, esperar 2s, 4s, 8s., etc.

 •   Cachear datos de catálogo agresivamente (ISR + Redis si escalamos).

 •   Considerar upgrade a cuenta developerpago de TMDb (50 req/s) a 10C/mes si usuarios crecen.

Fallos de infraestructura Vercel. Riesgo: Vercel podría experimentar downtime de su CDN o edge
functions, haciendo la aplicación inaccesible. Severidad: Alta. Probabilidad: Muy baja (SLA de
99.95 % típicamente honrado). Mitigación:

 •   Monitoreo externo mediante servicios de terceros (StatusPage, PagerDuty).

 •   Activar modo offline básico: cachear URLs visitadas recientemente en Service Worker.

 •   Plan de escalamiento: si downtime se normalizara, considerar multi-region deployment.


5.3.2 Riesgos legales y de privacidad

Términos de servicio de APIs. Riesgo: Los términos de servicio de TMDb, Trakt, etc., podrían
prohibir ciertos usos (ej. cacheado agresivo, reutilización de datos para entrenar modelos de IA,
monetización a través de afiliados). Severidad: Alta si se incumple. Probabilidad: Baja si se
mantiene conformidad. Mitigación:

 •   Revisar anualmente términos de servicio de cada API.

 •   Implementar opt-in explícito de usuarios para funcionalidades que podrían ser borderline (ej.
     análisis de datos para recomendaciones).


                                                   87
Capítulo 5. Conclusiones

 •   Documentar cumplimiento de ToS en archivo COMPLIANCE.md del repositorio.

RGPD y protección de datos. Riesgo: Los datos personales del usuario (historial, preferencias,
información de cuenta) están bajo regulación RGPD si la plataforma opera en UE. Incumplimiento
podría resultar en multas de hasta 4 % del turnover global. Severidad: Crítica. Probabilidad: Media
(aplicable si hay usuarios UE). Mitigación:

 •   Implementar política de privacidad clara y aceptación explícita al registrarse.

 •   Ofrecer funcionalidad de descarga de datos personales (GDPR Art. 20 - data portability).

 •   Implementar derecho al olvido: eliminar tokens y caché de usuario si lo solicita.

 •   Usar únicamente cookies httpOnly para tokens (no vulnerable a XSS).

 •   No compartir datos del usuario con terceros sin consentimiento explícito.

Copyright y derechos de autor. Riesgo: Mostrar imágenes (posters, backdrops) y metadata de
películas sin licencia explícita podría violar copyright. TMDb y estudios de cine podrían enviar
cease-and-desist. Severidad: Alta. Probabilidad: Baja (TMDb otorga uso no comercial libremente;
estudioss raramente demandan apps pequeñas). Mitigación:

 •   Usar únicamente imágenes de TMDb (licenciadas por TMDb bajo acuerdo con proveedores).

 •   Incluir atribución clara a TMDb, Trakt, IMDb en interfaces.

 •   En versión comercial, obtener licencias explícitas de estudio maledores o usar algún broker de
     licencias.

 •   No permitir descarga masiva de imágenes; usar solo visualización en-línea.


5.3.3 Riesgos comerciales

Competencia de servicios consolidados. Riesgo: Netflix, Disney+ u otros SVOD podrían lanzar
aplicación propia de tracking, relegando a The Show Verse a nicho. Severidad: Alta. Probabilidad:
Media. Mitigación:

 •   Diferenciación: mantener neutralidad respecto a plataformas de streaming, cubriendo todas por
     igual.

 •   Innovación continua: agregar features que ningún SVOD ofrece (ej. IA predictiva, análisis de
     patrones).

Capítulo 5. Conclusiones

 •   Comunidad: construir comunidad activa alrededor de usuario (foros, listas compartidas en
     Trakt).

 •   IP única: desarrollar componentes innovadores (grid de ratings, liquid button, análisis avanzado)
     que sean difíciles de copiar.

Fatiga de usuario con fragmentación. Riesgo: El usuario medio podría percibir The Show Verse
como «una app más» en un entorno ya saturado, sin motivación para cambiar de Trakt.tv u otro.
Severidad: Media. Probabilidad: Alta para adoptantes pasivos. Mitigación:

 •   UX diferenciado: interfaz moderna comparable a Netflix, no al «Trakt anticuado».

 •   Onboarding claro: mostrar en primeras pantallas qué problema único resuelve vs. competencia.

 •   Importación directa: permitir importar historial desde Trakt con un clic.

 •   Estarbuilding: SEO, marketing en comunidades de cine/series, menciones en blogs.


5.3.4 Síntesis de riesgos críticos y acciones prioritarias

Los riesgos que requieren atención inmediata son:


1) Cambios en APIs externas: Plan de monitoreo y graceful degradation (6 semanas de
     implementación).

2) Conformidad RGPD: Política de privacidad, cookie consent banner, derecho al olvido (4
     semanas).

3) Diferenciación de mercado: Roadmap de features únicas no disponibles en Trakt (negociación
     continua).


Con estas mitigaciones implementadas, el riesgo residual de The Show Verse es bajo-medio,
comparable al de startups tecnológicas similares en fase MVP.

móviles y habilitaría un modo de funcionamiento offline básico. El Service Worker podría cachear
las imágenes de pósters y los datos de catálogo para consulta sin conexión. Next.js 13+ soporta PWA
