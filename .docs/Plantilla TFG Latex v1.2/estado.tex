El presente estado de la cuestión aborda el contexto tecnológico y de mercado en el que se enmarca el proyecto. Se comienza con una visión del sector de las plataformas de \textit{streaming} y sus implicaciones para el usuario; a continuación se analizan las tecnologías de desarrollo web actuales que hacen posible la solución propuesta; y se finaliza examinando las plataformas existentes más relevantes.

\section{Marco teórico del trabajo}

\subsection{Las plataformas de streaming y sus consecuencias para el usuario}

La transición del modelo tradicional al \textit{streaming} bajo demanda se inició con el lanzamiento del servicio de Netflix en 2007, aunque su consolidación definitiva se produjo a partir de 2019, con la irrupción simultánea de Disney+, Apple TV+ y HBO Max. En 2024, el usuario medio en mercados como España o Estados Unidos tiene acceso simultáneo a entre 3 y 5 servicios de \textit{streaming} distintos, según datos de CNMC y Statista \parencite{CNMC:2025,Statista:2024}.

La distribución del contenido entre múltiples plataformas introduce varios problemas estructurales para el usuario. Explorar múltiples catálogos antes de decidir qué contenido consumir incrementa el tiempo de búsqueda y reduce la satisfacción percibida. En series con emisión semanal o consumidas de forma no continua, el usuario pierde fácilmente la referencia del último episodio visto, especialmente cuando gestiona múltiples series simultáneamente en diferentes plataformas. Las licencias temporales provocan que películas y series desaparezcan de una plataforma sin previo aviso o migren a otra, obligando al usuario a mantener múltiples suscripciones activas. Cada plataforma mantiene su propio registro de reproducción aislado, sin posibilidad de exportación o integración, impidiendo al usuario tener una visión global de su consumo audiovisual, generar estadísticas agregadas o recuperar su historial si cancela una suscripción. Finalmente, aspectos como la gestión de listas personalizadas, la valoración de contenido, las recomendaciones o el control parental presentan diferencias significativas de implementación entre servicios, dificultando la consistencia en la experiencia de usuario.

La respuesta a esta problemática ha sido el desarrollo de aplicaciones conocidas como \textit{media trackers} o plataformas de seguimiento de contenido audiovisual. Estas herramientas actúan como una capa de abstracción sobre las plataformas de \textit{streaming}, permitiendo al usuario mantener un registro unificado de su consumo independientemente del servicio donde visualice el contenido. Las funcionalidades típicas incluyen seguimiento de progreso por episodio, gestión de listas de pendientes y favoritos, historial completo de visualizaciones, estadísticas de consumo, descubrimiento de nuevos contenidos mediante recomendaciones y búsqueda de disponibilidad entre plataformas.

Este contexto de fragmentación y la necesidad de herramientas de gestión unificada constituye el problema fundamental que \textit{The Show Verse} pretende resolver mediante una solución técnicamente integrada que combine seguimiento, descubrimiento, información de disponibilidad y gestión de biblioteca local en una única aplicación web con una interfaz moderna y un rendimiento optimizado.

\subsection{Autenticación y APIs REST}

Se hace uso del estándar OAuth 2.0 \parencite{OWASP:2021} para la autenticación con Trakt.tv, que es el servicio que ofrece la funcionalidad de seguimiento más completa y que actúa como punto central de la gestión del contenido del usuario. TMDb también ofrece soporte para autenticación mediante OAuth 2.0, aunque en este caso se ha optado por un sistema de autenticación basado en API Key para simplificar la integración y evitar la complejidad adicional de gestionar tokens de acceso y renovación.

Las APIs utilizadas en este proyecto siguen el estilo REST con las siguientes características:

\textbf{TMDb API v3} \parencite{TMDb:2024} es una API REST pública con autenticación por API Key. Proporciona acceso a su base de datos de más de 800.000 películas y 150.000 series de televisión, con soporte para más de 40 idiomas. Su límite de peticiones es de 40 peticiones por 10 segundos para cuentas gratuitas.

\textbf{Trakt.tv API v2} \parencite{Trakt:2024} combina autenticación por API Key para \textit{endpoints} públicos con OAuth 2.0 Bearer Token para \textit{endpoints} autenticados. Un aspecto técnico relevante es el sistema de identificadores: Trakt mantiene sus propios slugs e IDs, pero acepta identificadores de IMDb y TMDb mediante el \textit{endpoint} de búsqueda por ID externo.

\textbf{OMDb API} \parencite{OMDb:2024} se utiliza como fuente complementaria para los metadatos y puntuaciones externas vinculadas al identificador de IMDb. Su integración se realiza mediante peticiones de lectura a través de la API interna del proyecto para unificar el formato de respuesta y control de errores.

\textbf{JustWatch} \parencite{JustWatch:2024} se integra para obtener enlaces de disponibilidad en plataformas de \textit{streaming}. En este caso no se consume una API REST pública oficial, sino que se encapsula la consulta en rutas internas del proyecto que normalizan parámetros, validan resultados y aplican caché antes de exponer el enlace en la interfaz.

\textbf{Plex Media Server} \parencite{Plex:2024} se incorpora como fuente del catálogo de contenido multimedia local del usuario. La comunicación se realiza por HTTP/HTTPS contra el servidor Plex configurado por el usuario y se protege mediante autenticación basada en token, lo que permite combinar en una misma vista contenido personal y datos procedentes de servicios en la nube.

\subsection{Rendimiento web}

El rendimiento de \textit{The Show Verse} se plantea de forma global, no solo para una vista concreta. La aplicación combina páginas con alta densidad de información, por lo que la estrategia prioriza mantener tiempos de respuesta estables cuando aumentan simultáneamente el número de elementos en pantalla y el volumen de llamadas a APIs \parencite{Google:2024a,Google:2024b}.

En la capa de renderizado se adopta un enfoque híbrido de SSR/ISR según la criticidad y frecuencia de cambio de los datos. Las vistas principales se sirven con el contenido inicial desde el servidor y revalidación periódica, mientras que los bloques secundarios se cargan de forma diferida en el cliente para no penalizar el primer renderizado. En paralelo, la agregación de datos se resuelve con \texttt{Promise.all} y límites de tiempo acotados, evitando que una dependencia externa lenta bloquee la página completa.

Este criterio se aplica tanto en los dashboards y la página de detalle como en las pantallas con listados extensos del usuario. En módulos como Historial, Favoritos, Pendientes y En progreso, la interfaz mantiene la fluidez mediante la combinación de filtrado y ordenación en cliente con cálculos memorizados, cachés de apoyo para puntuaciones e imágenes y reutilización de resultados cuando el usuario cambia entre modos de vista o vuelve a una sección ya consultada. Así se reduce trabajo repetido de renderizado y se evita recargar información ya resuelta.

Como criterio general, el proyecto guarda temporalmente información en varios niveles (servidor, API y navegador) para evitar repetir la misma consulta muchas veces en poco tiempo. Además, cuando varios componentes piden el mismo dato a la vez, la aplicación intenta resolverlo con una sola petición y reutilizar el resultado. En la práctica, esto reduce carga innecesaria sobre servicios externos con límites de uso, mejora la estabilidad cuando hay más tráfico y mantiene una navegación fluida en páginas con muchos elementos.

\subsection{Diseño de interfaz}

El diseño de interfaz de \textit{The Show Verse} utiliza un tema oscuro general para destacar los carteles (\textit{posters}) y fondos (\textit{backdrops}) del contenido teniendo como referencia las plataformas de \textit{streaming} actuales \parencite{Babich:2021,Nielsen:2024}.

\textbf{Vistas personalizadas.} 
Las principales secciones de la aplicación ofrecen múltiples modos de visualización: cuadrícula (\textit{grid}), lista (\textit{list}) o vista compacta (\textit{compact}). La elección del usuario se almacena en el navegador mediante \textit{localStorage} para mantener la preferencia en futuras visitas. Además, el Historial incluye una vista específica de calendario y la sección En progreso incorpora una vista de tarjetas con barras de porcentaje para mostrar el avance de las series.

\textbf{Diseño responsive.} 
La interfaz se adapta al dispositivo del usuario \parencite{MDN:2024d}. En la versión de escritorio, se utilizan efectos de animación al pasar el cursor (\textit{hover}) por las tarjetas para desplegar menús e información. En dispositivos móviles, estos efectos se desactivan y se sustituyen por toques directos en botones adaptados a pantallas táctiles evitando problemas de navegación.

\textbf{Componentes propios.} 
Para evitar depender de librerías genéricas y mantener una identidad visual coherente, se han desarrollado componentes personalizados:
\begin{itemize}
    \item \texttt{LiquidButton}: Botones de acción principales con diseño fluido y animado.
    \item \texttt{StarRating}: Sistema de valoración para el usuario.
    \item \texttt{DetailsSectionMenu}: Menú dinámico para organizar y navegar por las secciones dentro de la página de detalles de una película o serie.
    \item \texttt{ScoreboardBar}: Componente unificado que muestra las puntuaciones de las distintas plataformas externas así como los enlaces externos y una serie de estadísticas adicionales.
    \item \texttt{TraktWatchedModal} y \texttt{TraktEpisodesWatchedModal}: Ventanas emergentes propias para registrar directamente el visionado de películas y episodios de series en Trakt para cada usuario.
\end{itemize}

\textbf{Transiciones.} 
Se emplea la librería \textit{Framer Motion} \parencite{Framer:2024} para animar la interfaz al aplicar filtros, cambiar de vista o desplegar información lo que evita cambios de pantalla bruscos y mejora la experiencia de usuario.
