El desarrollo de \textit{The Show Verse} se ha abordado mediante una metodología iterativa basada en el ciclo de vida ágil, adaptada a un proyecto individual. La elección de este enfoque frente a las metodologías en cascada se justifica por la naturaleza exploratoria del proyecto ya que la integración con APIs de terceros implica incertidumbres técnicas que no siempre se pueden anticipar en una fase de análisis y la prioridad entre funcionalidades puede variar a medida que se comprende mejor el problema.

El proceso de desarrollo se realiza en iteraciones de aproximadamente una semana de duración. Al inicio de cada iteración se seleccionan del \textit{backlog} las funcionalidades a abordar, priorizadas según tres criterios:

\begin{itemize}
\item \textbf{Dependencia técnica:} las funcionalidades que sirven de base a otras (autenticación, cliente de API) se abordan primero.
\item \textbf{Valor aportado al usuario:} se priorizan las funcionalidades más visibles y relevantes para la experiencia de uso.
\item \textbf{Riesgo técnico:} las integraciones más inciertas se adelantan para conocer cuanto antes sus restricciones.
\end{itemize}

\section{Metodología}

\subsection{Criterios de integración de APIs}

La selección de las APIs externas ha seguido los siguientes criterios:

\begin{itemize}
\item TMDb fue seleccionada frente a otras alternativas por ofrecer la mayor cobertura de contenido, soporte en diferentes idiomas, galería de imágenes y datos de \textit{streaming}, además de disponer de la documentación más completa y una comunidad de desarrolladores activa.
\item Trakt.tv fue seleccionada como plataforma de sincronización por su API pública v2 bien documentada, su soporte completo de series a nivel de episodio y por ser la plataforma de \textit{tracking} más referenciada por la comunidad de desarrolladores independientes.
\item OMDb fue seleccionada únicamente para los datos de \textit{rating} de IMDb y Rotten Tomatoes, que TMDb no proporciona directamente.
\end{itemize}

\subsection{Criterios de decisión para la autenticación}

El problema de la autenticación OAuth 2.0 admitía varias opciones de implementación en Next.js. Se descartó el uso de librerías de autenticación genéricas como NextAuth.js / Auth.js por las siguientes razones:

\begin{itemize}
\item Trakt.tv no es un proveedor OAuth soportado de forma nativa por estas librerías.
\item La personalización necesaria (refresco de tokens específico de Trakt, intercambio de códigos con \textit{headers} personalizados) habría requerido más configuración que implementar el flujo directamente.
\item La implementación propia ofrece mayor control sobre el manejo de errores y el almacenamiento de tokens.
\end{itemize}

Se optó por implementar el flujo Authorization Code de forma manual mediante \textbf{API Routes de Next.js}, almacenando los tokens en cookies \texttt{httpOnly} con atributos \texttt{Secure} y \texttt{SameSite=Lax}. Esta decisión garantiza que los tokens no sean accesibles desde JavaScript del navegador (protección frente a XSS) y que no se envíen en peticiones \textit{cross-site} no intencionadas (protección parcial frente a CSRF).

\subsection{Estrategia de caché}

Para cada tipo de dato gestionado en la aplicación se ha definido una estrategia de caché en función de la frecuencia de actualización del dato en origen y su importancia para la experiencia de usuario.

\begin{table}[!htbp]
\begin{center}
\begin{tabular}{| l | l | l |}
\hline
\textbf{Tipo de dato} & \textbf{Almacenamiento} & \textbf{TTL} \\ 
\hline
\hline
Catálogo general (top rated, populares) & ISR servidor & 10 minutos \\
\hline
Detalles de película/serie & ISR servidor & 1 hora \\
\hline
Imágenes TMDb & CDN Vercel & 1 hora \\
\hline
Datos OMDb (proxy servidor) & Map en proceso & 24 h / 60 s \\
\hline
Datos OMDb en cliente & sessionStorage & 24 horas \\
\hline
Puntuaciones IMDb y Trakt en listas & localStorage & 30 días \\
\hline
Rating personal del usuario & Map en módulo & 10 min / 45 s \\
\hline
Estado personal del usuario & No-cache & En cada petición \\
\hline
\end{tabular}
\caption{Estrategia de caché por tipo de dato}
\label{tab:cache}
\end{center}
\end{table}

Esta diferenciación permite reducir el número de llamadas a APIs externas en un estimado del 70--80\% para los datos de catálogo, mientras se garantiza la actualización inmediata de los datos de usuario que requieren frescura absoluta.

\textbf{Caché en múltiples niveles:} el sistema implementa cuatro niveles de caché complementarios:

\begin{enumerate}
\item \textbf{ISR (Incremental Static Regeneration):} páginas estáticas regeneradas periódicamente en servidor, con servicio desde caché entre regeneraciones.
\item \textbf{Memory cache en proceso Node.js:} \texttt{Map} global que persiste durante la vida del proceso serverless (típicamente 5--15 min en Vercel), evitando peticiones redundantes a OMDb.
\item \textbf{sessionStorage en cliente:} caché de puntuaciones que persiste durante la sesión del navegador (hasta cerrar pestaña), común entre páginas.
\item \textbf{localStorage en cliente:} caché persistente de puntuaciones con TTL de 30 días, sobrevive entre sesiones del navegador.
\end{enumerate}

\textbf{Deduplicación en vuelo:} implementada en \texttt{lib/api/omdb.js}, si múltiples componentes solicitan el mismo dato simultáneamente, solo se realiza una petición HTTP y todas las solicitudes esperan el resultado compartido. Esto es crítico en listas de 100+ ítems donde varias tarjetas pueden necesitar el mismo \texttt{imdbId}.

\textbf{Pool de concurrencia:} para respetar los \textit{rate limits} de las APIs (OMDb: 1000 peticiones/día en plan gratuito), las peticiones paralelas se limitan a 4--5 simultáneas mediante un patrón de \textit{worker pool}. Cuando hay 100 peticiones pendientes, se ejecutan en lotes de 4, esperando a que una termine antes de lanzar la siguiente.

\textbf{Algoritmo LRU para localStorage:} al superar 500 entradas cacheadas, se aplica \textit{Least Recently Used} para eliminar entradas antiguas y mantener el tamaño del almacenamiento controlado (típicamente <2\,MB).

\section{Tecnologías empleadas}

\textbf{Next.js 16 (Framework principal).} Seleccionado frente a otras alternativas (Remix, SvelteKit, Nuxt 3) por ser el framework React con mayor adopción industrial (>40\% de cuota entre frameworks React según npm trends 2024), la mejor integración con Vercel y la documentación más extensa. El App Router de Next.js 13+ es la única opción que implementa React Server Components de forma estable y con soporte oficial.

\textbf{React 19.} La elección de React como biblioteca de UI es consecuencia directa de la elección de Next.js. React 19 introduce mejoras en el manejo de acciones de servidor y la promesa nativa en componentes de cliente.

\textbf{Tailwind CSS 4.} Seleccionado frente a CSS Modules, Styled Components o Emotion por el paradigma \textit{utility-first} que acelera el desarrollo sin sacrificar la personalización. La versión 4, basada en PostCSS con motor Rust nativo, elimina el PurgeCSS separado e incorpora mejoras sustanciales de rendimiento en el tiempo de compilación.

\textbf{Framer Motion 12.} La elección de Framer Motion frente a otras librerías de animación (React Spring, CSS Animations puras) se basa en su modelo declarativo basado en variantes, su soporte nativo para animaciones de \textit{layout} y su integración con el ciclo de vida de los componentes React mediante \texttt{AnimatePresence}.

\section{Gestión del desarrollo y control de versiones}

\subsection{Flujo de trabajo con GitHub}

El proyecto está alojado en un repositorio de GitHub con la siguiente estructura de ramas:

\begin{itemize}
\item \textbf{main:} rama de producción. Solo contiene código testeado y listo para desplegar. Se configura con protección de rama (require pull requests, require status checks).
\item \textbf{develop:} rama de integración. Recibe todos los pulls requests de desarrollo. Se despliega automáticamente a un entorno de staging.
\item \textbf{feature/*:} ramas de funcionalidades. Una rama por característica importante (feature/oauth-trakt, feature/plex-integration, etc.), ramificando desde \texttt{develop}.
\end{itemize}

\section{Proceso de despliegue en producción}

\subsection{Plataforma de hosting (Vercel)}

El proyecto se despliega en Vercel, la plataforma de hosting optimizada para Next.js como autor del framework. Las razones de esta selección son:

\begin{itemize}
\item \textbf{Integración native con Next.js:} Las características de Next.js como Image Optimization, ISR y Edge Functions se soportan de forma nativa sin configuración adicional.
\item \textbf{API serverless:} Las API Routes se auto-escalan sin gestionar servidores o contenedores.
\item \textbf{Edge computing:} Los Edge Functions ejecutan código más cerca del usuario, reduciendo latencia para SPA middleware.
\item \textbf{Deployments automáticos:} Cada pull request genera un URL único de vista previa con el código del branch.
\item \textbf{Git-based deploys:} El despliegue se dispara automáticamente al hacer push o merge a una rama configurada.
\item \textbf{Monitoreo integrado:} Web Analytics, Performance Monitoring y Error Tracking nativos.
\end{itemize}

\subsection{Variables de entorno y credenciales}

Para garantizar la seguridad de la aplicación y evitar exponer información confidencial en el código fuente, todas las claves y contraseñas de las APIs se centralizan mediante el administrador de variables de entorno cifradas que integran de forma nativa las plataformas de GitHub y Vercel:

\begin{itemize}
\item \texttt{NEXT\_PUBLIC\_TMDB\_API\_KEY}: Clave de la API de TMDb, accesible desde el cliente y servidor.
\item \texttt{OMDB\_API\_KEY}: Clave privada de acceso a la base de datos de OMDb.
\item \texttt{TRAKT\_CLIENT\_ID} y \texttt{TRAKT\_CLIENT\_SECRET}: Credenciales principal y secreta para autenticar la aplicación como cliente de Trakt.tv.
\item \texttt{TRAKT\_REDIRECT\_URI}: Dirección web de retorno requerida en el flujo de autenticación OAuth de Trakt.
\item \texttt{PLEX\_URL} y \texttt{PLEX\_TOKEN}: Dirección de red (local o externa) y token personal del servidor multimedia de Plex.
\item \texttt{PLEX\_CLIENT\_IDENTIFIER}: Identificador único de aplicación para Plex.
\item \texttt{PLEX\_JWT\_SCOPE} y \texttt{PLEX\_JWT\_KID}: Conjunto de permisos y nombre de clave asociados a la generación de JWT para Plex.
\item \texttt{PLEX\_JWT\_PRIVATE\_KEY}: Clave privada empleada en el servidor para la comunicación segura con Plex.
\end{itemize}

\subsection{Proceso de deployment}

Un despliegue típico a producción sigue estos pasos:

\begin{enumerate}
\item Se realiza merge de PR a \texttt{main} desde GitHub.
\item GitHub webhook notifica a Vercel del cambio.
\item Vercel inicia el proceso de build: instala dependencias, ejecuta \texttt{next build}, genera artefactos de producción.
\item Build check de Vercel valida que no hay errores de compilación.
\item Si el build es exitoso, Vercel despliega a infraestructura de producción.
\item Vercel ejecuta health checks (petición HTTP a raíz de dominio) para verificar que la aplicación responde.
\item Dominio \texttt{the-show-verse.vercel.app} apunta automáticamente a la versión nueva.
\item Si hay error, Vercel realiza rollback automático a la versión anterior.
\item URL de deployment anterior permanece disponible como \texttt{previousUrl.vercel.app} para rollback manual.
\end{enumerate}
