El desarrollo de \textit{The Show Verse} se ha abordado mediante una metodología iterativa basada en el ciclo de vida ágil, adaptada a un proyecto individual. La elección de este enfoque frente a las metodologías en cascada se justifica por la naturaleza exploratoria del proyecto ya que la integración con APIs de terceros implica incertidumbres técnicas que no siempre se pueden anticipar en una fase de análisis y la prioridad entre funcionalidades puede variar a medida que se comprende mejor el problema.

El proceso de desarrollo se realiza en iteraciones de aproximadamente una semana de duración. Al inicio de cada iteración se seleccionan del \textit{backlog} las funcionalidades a abordar, priorizadas según tres criterios:

\begin{itemize}
\item \textbf{Dependencia técnica:} las funcionalidades que sirven de base a otras (autenticación, cliente de API) se abordan primero.
\item \textbf{Valor aportado al usuario:} se priorizan las funcionalidades más visibles y relevantes para la experiencia de uso.
\item \textbf{Riesgo técnico:} las integraciones más inciertas se adelantan para conocer cuanto antes sus restricciones.
\end{itemize}

\section{Metodología}

\subsection{Criterios de integración de APIs}

La selección de las APIs externas ha seguido los siguientes criterios:

\begin{itemize}
\item TMDb fue seleccionada frente a otras alternativas por ofrecer la mayor cobertura de contenido, soporte en diferentes idiomas, galería de imágenes y datos de \textit{streaming}, además de disponer de la documentación más completa y una comunidad de desarrolladores activa \parencite{TMDb:2024}.
\item Trakt.tv fue seleccionada como plataforma de sincronización por su API pública v2 bien documentada, su soporte completo de series a nivel de episodio y por ser la plataforma de \textit{tracking} más referenciada por la comunidad de desarrolladores independientes \parencite{Trakt:2024}.
\item OMDb fue seleccionada únicamente para los datos de \textit{rating} de IMDb y Rotten Tomatoes, que TMDb no proporciona directamente \parencite{OMDb:2024}.
\end{itemize}

\subsection{Criterios de decisión para la autenticación}

El problema de la autenticación OAuth 2.0 admitía varias opciones de implementación en Next.js. Se descartó el uso de librerías de autenticación genéricas como NextAuth.js / Auth.js por las siguientes razones:

\begin{itemize}
\item Trakt.tv no es un proveedor OAuth soportado de forma nativa por estas librerías.
\item La personalización necesaria (refresco de tokens específico de Trakt, intercambio de códigos con \textit{headers} personalizados) habría requerido más configuración que implementar el flujo directamente.
\item La implementación propia ofrece mayor control sobre el manejo de errores y el almacenamiento de tokens.
\end{itemize}

Se optó por implementar el flujo Authorization Code de forma manual mediante \textbf{API Routes de Next.js}, almacenando los tokens en cookies \texttt{httpOnly} con atributos \texttt{Secure} y \texttt{SameSite=Lax} \parencite{NextJS:2024,OWASP:2021}. Esta decisión garantiza que los tokens no sean accesibles desde JavaScript del navegador (protección frente a XSS) y que no se envíen en peticiones \textit{cross-site} no intencionadas (protección parcial frente a CSRF).

\subsection{Estrategia de caché}

Para cada tipo de dato gestionado en la aplicación se ha definido una estrategia de caché en función de la frecuencia de actualización del dato en origen y su importancia para la experiencia de usuario. En el caso de las consultas a OMDb en el proxy servidor, las respuestas exitosas se guardan durante 24 horas y se reutilizan en solicitudes posteriores, mientras que los errores temporales se almacenan solo 60 segundos para evitar insistir de forma inmediata y permitir un nuevo intento poco después.

Esta diferenciación permite reducir el número de llamadas a APIs externas en un estimado del 70--80\% para los datos de catálogo, mientras se garantiza la actualización inmediata de los datos de usuario que requieren frescura absoluta \parencite{MDN:2024a,Vercel:2024c}.

\textbf{Caché en múltiples niveles:} el sistema implementa cuatro niveles de caché complementarios \parencite{MDN:2024a}:

\begin{enumerate}
\item \textbf{ISR (Incremental Static Regeneration):} páginas estáticas regeneradas periódicamente en servidor, con servicio desde caché entre regeneraciones.
\item \textbf{Memory cache en proceso Node.js:} \texttt{Map} global que persiste durante la vida del proceso serverless (típicamente 5--15 min en Vercel), evitando peticiones redundantes a OMDb.
\item \textbf{sessionStorage en cliente:} caché de puntuaciones que persiste durante la sesión del navegador (hasta cerrar pestaña), común entre páginas.
\item \textbf{localStorage en cliente:} caché persistente de puntuaciones con TTL de 30 días, sobrevive entre sesiones del navegador.
\end{enumerate}

\textbf{Deduplicación en vuelo:} implementada en \texttt{lib/api/omdb.js}, si múltiples componentes solicitan el mismo dato simultáneamente, solo se realiza una petición HTTP y todas las solicitudes esperan el resultado compartido. Esto es crítico en listas de 100+ ítems donde varias tarjetas pueden necesitar el mismo \texttt{imdbId}.

\textbf{Pool de concurrencia:} para respetar los \textit{rate limits} de las APIs (OMDb: 1000 peticiones/día en plan gratuito), las peticiones paralelas se limitan a 4--5 simultáneas mediante un patrón de \textit{worker pool}. Cuando hay 100 peticiones pendientes, se ejecutan en lotes de 4, esperando a que una termine antes de lanzar la siguiente.

\textbf{Algoritmo LRU para localStorage:} al superar 500 entradas cacheadas, se aplica \textit{Least Recently Used} para eliminar entradas antiguas y mantener el tamaño del almacenamiento controlado (típicamente <2\,MB).

\section{Tecnologías empleadas}

\textbf{Next.js 16 (Framework principal).} Seleccionado frente a otras alternativas (Remix, SvelteKit, Nuxt 3) por ser el framework React con mayor adopción industrial (>40\% de cuota entre frameworks React según npm trends 2024), la mejor integración con Vercel y la documentación más extensa \parencite{Wieruch:2023}. El App Router de Next.js 13+ es la única opción que implementa React Server Components de forma estable y con soporte oficial \parencite{Vercel:2024a,Meta:2024}.

\textbf{React 19.} La elección de React como biblioteca de UI es consecuencia directa de la elección de Next.js. React 19 introduce mejoras en el manejo de acciones de servidor y la promesa nativa en componentes de cliente \parencite{React:2024}.

\textbf{Tailwind CSS 4.} Seleccionado frente a CSS Modules, Styled Components o Emotion por el paradigma \textit{utility-first} que acelera el desarrollo sin sacrificar la personalización. La versión 4, basada en PostCSS con motor Rust nativo, elimina el PurgeCSS separado e incorpora mejoras sustanciales de rendimiento en el tiempo de compilación \parencite{TailwindCSS:2024}.

\textbf{Framer Motion 12.} La elección de Framer Motion frente a otras librerías de animación (React Spring, CSS Animations puras) se basa en su modelo declarativo basado en variantes, su soporte nativo para animaciones de \textit{layout} y su integración con el ciclo de vida de los componentes React mediante \texttt{AnimatePresence} \parencite{Framer:2024}.

\section{Gestión del desarrollo y control de versiones}

\subsection{Flujo de trabajo con GitHub}

El proyecto está alojado en un repositorio de GitHub con la siguiente estructura de ramas:

\begin{itemize}
\item \textbf{main:} rama de producción. Solo contiene código testeado y listo para desplegar. Se configura con protección de rama (require pull requests, require status checks).
\item \textbf{develop:} rama de integración. Recibe todos los pulls requests de desarrollo. Se despliega automáticamente a un entorno de staging.
\item \textbf{feature/*:} ramas de funcionalidades. Una rama por característica importante (feature/oauth-trakt, feature/plex-integration, etc.), ramificando desde \texttt{develop}.
\end{itemize}

\section{Proceso de despliegue en producción}

El proyecto ha pasado por dos fases de despliegue a lo largo de su desarrollo. En la fase inicial se utilizó Vercel como plataforma de hosting gestionada en la nube. Posteriormente, la aplicación se migró a una infraestructura propia basada en un servidor NAS con acceso público a través de Cloudflare Tunnel, que constituye la configuración actualmente en producción.

\subsection{Fase inicial: plataforma Vercel}

Durante las primeras fases del desarrollo la aplicación se desplegó en Vercel \parencite{Vercel:2024a,Vercel:2024d}, una plataforma de hosting optimizada para Next.js. Las razones de esta selección inicial son:

\begin{itemize}
\item \textbf{Integración nativa con Next.js:} Las características de Next.js como Image Optimization, ISR \parencite{Vercel:2024c} y Edge Functions \parencite{Vercel:2024b} se soportan de forma nativa sin configuración adicional.
\item \textbf{API serverless:} Las API Routes se auto-escalan sin gestionar servidores o contenedores.
\item \textbf{Edge computing:} Los Edge Functions ejecutan código más cerca del usuario, reduciendo latencia para SPA middleware.
\item \textbf{Deployments automáticos:} Cada pull request genera un URL único de vista previa con el código del branch.
\item \textbf{Git-based deploys:} El despliegue se dispara automáticamente al hacer push o merge a una rama configurada.
\item \textbf{Monitoreo integrado:} Web Analytics, Performance Monitoring y Error Tracking nativos.
\end{itemize}

\subsection{Proceso de deployment en Vercel}

El despliegue se ejecuta automáticamente cuando se hace merge a la rama \texttt{main} en GitHub. Vercel recibe la notificación, instala dependencias, ejecuta \texttt{next build} y valida que no haya errores de compilación. Si el build es exitoso, despliega la nueva versión y actualiza el dominio, en caso de fallo, realiza un rollback automático a la versión anterior.

\subsection{Despliegue actual: servidor NAS propio}

La aplicación está desplegada actualmentese en un servidor propio bajo el dominio \texttt{theshowverse.com}, mediante tres componentes principales. La aplicación se empaqueta en un contenedor Docker \parencite{Docker:2024} construido con un \texttt{Dockerfile} multietapa que separa la fase de compilación de la de ejecución, reduciendo el tamaño final de la imagen. Las variables de entorno sensibles se inyectan en tiempo de ejecución mediante un archivo \texttt{.env} externo al repositorio. El acceso público se gestiona a través de Cloudflare Tunnel \parencite{Cloudflare:2024}: un segundo contenedor ejecuta \textit{cloudflared}, que mantiene una conexión cifrada saliente hacia Cloudflare, enrutando el tráfico del dominio al puerto \texttt{3000} del servidor sin necesidad de exponer puertos en el router ni disponer de IP pública fija. Por último, existe una página de mantenimiento para cuando el servidor no está disponible mediante un Cloudflare Worker.

\subsection{GitHub Actions}

Los despliegues en producción se automatizan mediante un \textit{workflow} de GitHub Actions. Cuando se hace \textit{push} a la rama principal del repositorio, el \textit{workflow} se conecta por SSH al NAS, descarga la nueva versión del código, reconstruye la imagen Docker y reinicia el contenedor de la aplicación. De esta forma, cada cambio confirmado en el repositorio se refleja en producción sin intervención manual.
